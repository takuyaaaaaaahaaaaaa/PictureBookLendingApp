# デザイン原則（DESIGN PRINCIPLES）

このドキュメントは、UI/UXに関する意思決定の「物差し」です。
新しい画面・機能・issueを評価するときは、まずここに照らして判断します。

関連ドキュメント: [SCREEN_DESIGN.md](SCREEN_DESIGN.md) / [TERMS.md](TERMS.md)

---

## 1. 利用シーン定義（最重要）

### 主シーン：お迎えラッシュの15分間

アプリの真価が問われるのは、静かな事務室ではなく **夕方のお迎え時間帯** です。

> 夕方17時。保育室の入り口に置かれた iPad の前に、絵本を返しに来た親子と
> 借りに来た親子が次々とやってくる。先生は片手で園児の手を引きながら、
> もう片手で操作する。1組あたりに使える時間は **30秒以内**。
> 後ろには次の親子が待っている。

このシーンから導かれる設計基準：

| 基準 | 数値目標 | 理由 |
|---|---|---|
| 貸出完了までのタップ数 | **3タップ以内** | 行列を作らない |
| 返却完了までのタップ数 | **2タップ以内** | 返却は貸出より頻度が高い |
| 操作の取り消し | **1タップで直前の操作を戻せる** | 急いでいると必ず押し間違える |
| 文字・ボタンサイズ | 1m離れて読める / 片手親指で押せる | 抱っこしながら操作する |
| フィードバック | 操作結果が音・触覚・視覚で即わかる | 画面を凝視できない状況 |
| 待ち時間 | 全操作1秒以内レスポンス | 非機能要件と同じ |

### 副シーン：すきま時間の管理作業

昼寝の時間や閉園後に、先生が落ち着いて行う作業。

> 新しく届いた絵本20冊をまとめて登録する。月末に貸出状況を眺めて、
> ずっと借りられていない絵本や、返却期限を過ぎている利用者を確認する。

このシーンから導かれる設計基準：

- **一括操作**ができること（1冊ずつ20回繰り返させない）
- 登録は**カメラ・画像認識でショートカット**できること
- 貸出状況は**組ごと・期限ごとに俯瞰**できること
- 速さより**確実さ・わかりやすさ**を優先してよい

### ペルソナ（軽量版）

| | 主役：先生 | 脇役：保護者 |
|---|---|---|
| ITリテラシー | スマホは使うがITは本業でない | 様々 |
| 状況 | 常に忙しい・両手が塞がりがち | 子どもを急かしている |
| アプリへの期待 | 「考えなくても使える」 | 操作はほぼしない（見るだけ） |
| 最悪の体験 | 行列の前で操作に迷う・ミスが戻せない | 待たされる |

## 2. トップタスクと画面の一等地

操作頻度に応じて、画面の「一等地」（タブ直下・ファーストビュー・大きなボタン）を割り当てます。

| 頻度 | タスク | 一等地の優先度 |
|---|---|---|
| 毎日数十回 | 貸出・返却 | ★★★ 最優先。ホームから迷わず到達 |
| 毎日数回 | 絵本・利用者の検索 | ★★ 検索バーは常時見える位置 |
| 月数回 | 絵本・利用者の登録 | ★ 設定の奥でよいが、一括操作必須 |
| 年数回 | 組の編成・貸出設定 | 設定の奥でよい |

**原則：頻度の低いタスクのために、頻度の高いタスクのタップ数を増やさない。**

## 3. デザイン原則（判断に迷ったらここに戻る）

1. **行列ファースト** — すべてのUI判断は「後ろに親子が並んでいても大丈夫か？」で評価する
2. **ミスは前提、リカバリーは1タップ** — 確認ダイアログを増やすより、取り消しを用意する
3. **画面を見なくてもわかる** — ハプティクス・サウンド・大きな状態変化で結果を伝える
4. **登録は楽をさせる** — カメラ・画像認識・一括操作で「手入力」を最後の手段にする
5. **用語は TERMS.md に従う** — 「絵本」「利用者」「組」。先生が日常で使う言葉だけを使う
6. **HIG・Liquid Glass 準拠** — 独自UIを発明する前に、iOS標準の作法に乗る

## 4. フィードバック設計（成功は軽く・節目は楽しく・エラーは重く）

「フィードバックを消す」のではなく「**確認のタップを取り除く**」。
結果は必ず画面に残し、ユーザーを止めるのはエラーのときだけ。

| 種類 | 頻度 | 表現 | 必要タップ |
|---|---|---|---|
| 日常の成功（貸出・返却） | 毎日数十回 | 状態変化（ボタンが貸出⇄返却に切替・一覧に反映）＋チェックマーク表示（1.5秒で自動消滅）＋ハプティクス（.success） | 0 |
| 取り消し可能な操作（返却など） | 毎日数十回 | 「元に戻す」付きフィードバックカード（画面中央・2秒で自動消滅。取り消しボタンは控えめなスタイルにし、反射タップによる誤取り消しを防ぐ） | 0（戻すときだけ1） |
| 節目のお祝い（10冊・20冊・100冊…） | 月数回 | 紙吹雪アニメーション＋「○○ちゃん、100冊おめでとう！」（3〜4秒で自動終了・画面タップでスキップ可） | 0 |
| エラー | まれ | モーダルアラート（OKタップ必須）。**唯一、操作を止めてよい** | 1 |

### 設計ルール

- **アラート（OKボタン）を日常の成功に使わない。** 毎日数十回の操作では人はOKを反射で
  連打するようになり、本当に読むべきエラーアラートまで素通りされる。
  アラートをエラー専用にすることで、エラーの注意喚起力を守る
- **安心の根拠は「状態が画面に残ること」。** トーストや演出は消えても、
  貸出管理タブ・利用者カードに結果が反映されていて、いつでも確認できる
- **お祝いは盛ってよいが、行列は止めない。** 節目は特別な瞬間なので日常より大きな演出に
  する（主役は園児。お迎えの場で先生・保護者・本人が一緒に見られる）。
  ただし自動終了＋タップスキップは必須で、誰かのOKタップを待たない
- **お祝いは消えても残す。** 利用者カードに「100冊よんだよ」バッジとして記録し、
  あとから本人・保護者が見返せる資産にする

## 5. タイポグラフィ（文字サイズ方針）

方針：**主動線はデフォルトで大きく＋Dynamic Typeを壊さない。アプリ内の文字サイズ設定は作らない。**

背景：保育室のiPadは**共用端末**。Dynamic Typeは個人の設定なので、
「見えにくい人が自分で設定を変える」モデルが成立しない（先生もお年寄りも同じ画面を見る）。
したがって主動線は誰の設定にも頼らず読めるサイズを基準にする。

| 用途 | スケール | 根拠 |
|---|---|---|
| 主動線の本文（絵本タイトル・利用者名） | `.title3` 相当以上 | 1m離れて読める（利用シーン基準） |
| 状態・結果表示（フィードバックカード等） | `.title3`〜`.title` 相当の太字 | 画面を凝視できない状況でも読める |
| 補足情報（著者名・管理番号・期限等） | `.subheadline` 以上 | |
| ボタン | ラベル `.body` 以上・タップ領域44pt以上 | 片手親指で押せる |
| 管理系画面（設定・登録フォーム） | 標準サイズ | すきま時間に使う画面。Dynamic Type追従で十分 |

### 実装ルール

- **semantic font**（`.title3` 等）を使い、`Font.system(size:)` の固定pt指定を避ける。
  アイコン等どうしても固定値が要る箇所は `@ScaledMetric` でスケールさせる
- 大きい文字でも `lineLimit` で切り捨てず、折り返しで読めるレイアウトにする
- 「もっと大きくしたい」園には、iOS標準の**アプリごとの文字サイズ**
  （設定 → アクセシビリティ → アプリごとの設定）を案内する。アプリ内設定は作らない
- 検証は Preview の `.dynamicTypeSize(.accessibility3)` 程度まで崩れないことを確認する
