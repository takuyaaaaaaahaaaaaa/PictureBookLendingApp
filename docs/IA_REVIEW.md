## 2. 構造的な発見：動線がすべて「絵本起点」

現行IAは2タブとも **絵本→利用者** の順で操作が流れる。
しかしお迎えラッシュの実態は **親子（利用者）単位** でiPadの前に来る：

> 「Aちゃんが2冊返して、1冊借りる」

これを現行IAでやると：返却3タップ×2冊 + 絵本を探して貸出7タップ = **13タップ＋検索2回**。
主シーン（1組30秒以内）に対して、IAの主語が逆向きになっている。

## 3. 再設計案

### Phase 1：タップ削減（IA据え置き・即効性重視）

| 施策 | 効果 | 関連issue |
|---|---|---|
| 成功アラート廃止 → ハプティクス＋自動クローズ | 貸出・返却とも−1タップ | #18 |
| 返却の確認ダイアログ → 「元に戻す」スナックバーに置換 | 返却 3→**1タップ**＋Undo | #38 |
| 貸出フォームが前回選択した組を記憶 | 連続貸出時 −2タップ（※仮説検証中） | — |
| `.searchable` 二重定義の解消 | 検索・ソート不具合の解消 | #143 |

**Phase 1後：貸出 7→4タップ、返却 3→1タップ**（原則2「ミスは前提、リカバリーは1タップ」準拠）

> **※「組の記憶」は仮説検証中の機能。** 「お迎えは組ごとにまとまって来る」という
> 推測に基づく。実際にはきょうだいで別の組・お迎えがバラバラ等で組が連続しない
> 可能性があり、効果が出る頻度は未検証。ただし外れても無害（前回の組が選択済みに
> なるだけで利用者は未選択＝誤操作にならない）なため採用。Phase 2で利用者起点IAに
> 作り替える際、貸出フォーム自体が消える可能性があるので、そのとき残否を再判断する。

### Phase 2：利用者起点の動線を追加（IAの再構成）

> **⚠️ この節の具体案（組グリッド等）はインタビュー証拠により置き換え済み。**
> 最新の確定設計は [SCREEN_DESIGN_PHASE2.md](SCREEN_DESIGN_PHASE2.md)、経緯は §6 を参照。
> 以下は当初案の記録として残す。

タブ構成を「絵本」「貸出管理」から **「絵本」「利用者」** に再編する：

```
└ 利用者タブ（現・貸出管理タブを拡張）
  ├ 組の大ボタングリッド（もも組・ばら組…）…… 1タップ
  ├ 利用者一覧（あいうえお順・大きな行）……… 1タップ
  └ 利用者カード画面
    ├ 借りている絵本一覧 → 各行「返却」……… 1タップ/冊
    └ 「新しく借りる」→ 絵本選択 → 完了 …… 2タップ/冊
```

- 「Aちゃんが2冊返して1冊借りる」= 組1＋名前1＋返却2＋貸出2 = **6タップ**（現行13）
- 利用者を選んだ時点で組・名前の入力が不要になるため、貸出フォーム自体が消える
- 延滞バッジ・期限表示は利用者カードに集約（副シーンの俯瞰は組グリッドで担保）
- 絵本起点の動線（絵本タブ）は維持。両起点が揃って初めて現場の流れに合う

- スリープモード（#117）：常設iPad前提の運用に合わせる

## 4. 音声入力の検討（WWDC26時点）

対象環境：11インチ iPad Air (M3)・iOS 27視野。

| 手段 | オフライン | 評価 |
|---|---|---|
| キーボードのマイク（音声入力） | ✅ | ゼロコストだが削減効果小 |
| Speech framework（iOS 26〜 SpeechAnalyzer/SpeechTranscriber） | ✅ 完全オンデバイス | 利用者200人・絵本500冊の**閉じた語彙**へのマッチングなので精度を稼ぎやすい |
| FoundationModels（iOS 27で音声理解対応・OS同梱） | ✅ 完全オンデバイス | 発話→`{組, 利用者, 絵本, アクション}` の構造化まで可能。M3なら性能十分 |
| Siri / App Intents（SiriKitは非推奨化） | △ intentごとにオンデバイス必須を宣言可 | クラウドルーティング経路は採用しない（オフライン要件・園児名のプライバシー） |

**方針：音声は主動線ではなく「検索の加速装置」。**

- お迎えラッシュの騒音（子どもの声）下では誤認識→言い直しの「音声渋滞」リスクがあり、
  行列ファースト原則により主動線はタップを維持
- Phase 2の利用者起点IAにマイクボタンを重ねる：「もも組のたろうくん」→利用者カード直行→確認1タップ
- 同名対策：認識結果は必ず候補表示し1タップで確定。実行前確認＋Undoは音声でも省略しない
- 園児の名前を扱うため音声処理は**オンデバイス限定**（クラウド送信不可）

### 学び（IA設計の進め方）

IAが前進したのは推論を重ねたときではなく、**証拠が入ったとき**だった
（「1人1冊」の現場知識、インタビューのプライバシー懸念）。以後のIA変更は
「どの証拠に基づくか」を必ずこの表に追記してから行う。

### 追記：動詞ファーストへの転換（2026-06-16 同日）

上記の決定後、さらに以下の証拠で設計を更新した：

- **現場知識（オーナー）**：「借りる」と「返す」は本選びの時間を挟んだ**別々の用事**であり、
  同一画面に返却と貸出を同居させる利点が薄い。返す動線に無借用者を表示する意味もない
- **共用端末の定石**：ATM・券売機と同じキオスク型利用のため「最初に用事を選ぶ」動詞ファーストが適合
- **切替容易性（オーナー指摘）**：ホーム大ボタンよりタブの方が用事の行き来が常時1タップで速い

→ **タブ構成を「かえす（既定）／かりる／図書」に決定**。
「名前のみ一覧＋子の画面」のプライバシー設計は各動詞タブ内の部品として維持。
確定設計は [SCREEN_DESIGN_PHASE2.md](SCREEN_DESIGN_PHASE2.md)。

### 追記2：かりるの本起点化とタブ実装方針（2026-06-16 同日）

- **現場知識（オーナー）**：借りる人は棚で選んだ本を**手に持って**iPadに来る。
  また園児の枠が満杯でも保護者の枠で借りるケースがあり、枠の都合は最後にしか分からない
  → **かりるは本起点（本→名前→枠）に変更**。枠タップが実行確認を兼ねる
- 副作用として図書タブが不要になり（かりるタブの起点＝図書一覧が兼務）、**タブは2つ**に
- **タブバー実装（オーナー指摘：Liquid Glassはデフォルトに合わせたい）**：
  標準TabView（上部タブ）＋文字サイズ底上げで開始し、ペーパーテストで田中さん基準に
  不足と判明した場合のみコンテンツ内の大型切替（標準部品）にフォールバック。
  フルカスタムボトムバーは最終手段。**逸脱を先回りしない**

### 追記3：切替UIの確定（2026-06-16 同日）

- TabViewは使わず、**コンテンツ上部の大型切替（かえす／かりる）**に確定
- 根拠：発見7「一番触ってほしいところ大きく」／標準タブバーはDynamic Typeで育たない／
  **視線導線は上から下（オーナー指摘）**なので切替は一覧より上に置く
- タブバーの模造（カスタムボトムバー）はLiquid Glass逸脱のため不採用。
  タブバー非搭載＋標準部品の切替はHIG上合法で、逸脱にあたらない
- 戻り先：ペーパーテストで大型切替が過剰と分かれば標準TabViewへ

### 追記4：切替UIは標準TabViewに決定（2026-06-16 同日・最終）

- 大型切替（C案）はいったん採用したが、**未検証の推論（発見7の適用）のために
  システムタブバーの無料の挙動（表示・非表示・質感・状態管理）を捨てて
  自前管理コストを払うのは「逸脱を先回りしない」方針に反する**（オーナー判断）ため撤回
- **標準TabView＋「アプリごとの文字サイズ」最大化で開始**。既定タブ＝かえすなので
  タブ操作の頻度自体が低い
- ペーパーテスト・実機で「67歳基準で小さすぎる」証拠が出た場合のみC案（コンテンツ上部の
  大型切替）へ移行。A⇄Cは中身の2画面が共通で移行コストが低い

### 追記5：かえす一覧に保護者名も表示（2026-06-16 同日）

- **オーナー指摘**：親の名前も一覧に出してよいのでは
- 検討の結果採用。「子の名前だけ一覧」には、保護者が自分の借りた本を
  **自分の名前で探して見つからない**という隠れた失敗モードがあった
- `UserType.guardian(relatedChildId:)` の紐付けをコードで確認し、
  **どの家族名から入っても同じ「家庭の画面」（2枠）に着地**する形に統合。
  間違うドアがなく、家族2冊も1回の入場で返せる

### 追記6：俯瞰画面は作らず、かえすタブが兼ねる（2026-06-16 同日）

- **オーナー指摘**：先生の俯瞰もかえすタブでよいのでは
- 論点は「延滞している家の名前が他の保護者にも見えること」の許容可否だった
- **相反する証拠を両方記録する**：
  - 発見5（インタビュー）：「見えるのが嫌だなって話もちらっとあった」
  - 現場知識（オーナー）：園の**入園管理では名前一覧が全保護者に見える運用が既にある**。
    現行の絵本管理（貸出管理タブ）も誰が何を借りているか可視のまま、問題なく運用されている実績
- オーナー判断で**許容**。かえすタブに「きげんすぎ」マーク＋「きげんすぎのみ」フィルタを置き、
  **「設定 > 貸出状況」は作らない**（画面が1つ減る）。
  ただし書名・期限は一覧に出さず家庭の画面まで（発見5への配慮はこの線で維持）
- 将来「なぜ延滞が見えるのか」と迷ったらこの記録に戻ること

### 追記7：HIG確認と延滞数タブバッジの廃止（2026-06-16 同日）

- **オーナーの不安**：「HIGではタブはタスクではなくオブジェクトがよいという話がなかったか」
- HIG原文（Tab bars）を確認。禁止は「アクションを実行するタブ」であり、
  機能セクションを表すタブは合法（公式例＝時計アプリのアラーム/ストップウォッチ/タイマー）。
  「かえす/かりる」はセクション表示のナビゲーションなので**HIG準拠**と結論
- 原文確認の副産物として、HIG "Reserve badges for critical information" に照らし
  **現行の延滞数タブバッジ（`.badge(overDueCount)`）をPhase 2で廃止**と決定。理由3点：
  1. **常時点灯になる**：200家庭運用で延滞ゼロの瞬間はほぼなく、点きっぱなしの警告は
     情報量ゼロ（HIGの言う impact の希釈）
  2. **宛先と表示先のズレ**：バッジの宛先は管理担当の先生1人だが、iPadの前に立つのは
     9割が保護者。他家の延滞集計は保護者には行動可能性ゼロ
  3. **集計値は自分事にならない**：延滞中の家庭が見ても「園全体で7件」からは気づけない
- 先生向けの受動通知が将来必要になったら、正しいチャネルは **#132（Widgetに貸出数表示）**
  （先生の見る場所に先生宛の情報を出す＝宛先と表示先が一致する）

### 追記8：UI文言はひらがなにしない（2026-06-16 同日）

- **前提修正（オーナー）**：端末を操作するのは**保護者のみ**。園児は操作しない
- したがってUI文言を園児向けにひらがな化する意味はなく、むしろ大人（67歳基準含む）には
  ひらがなの連続は漢字表記より読み取りが遅い。**通常の大人向け表記に統一**：
  - タブ名：「返却」「貸出」（図書館用語として既知・TERMS準拠・発見4の粒度も名詞で統一）
  - フィルタ：「延滞のみ」／空状態：「現在、貸出中の利用者はいません」
  - 枠名：「園児の本」「保護者の本」
- **園児の名前がひらがななのはデータ登録の運用**（園ではすべてひらがな入力）であり、
  UI文言の話とは別。名前表示はデータのまま
